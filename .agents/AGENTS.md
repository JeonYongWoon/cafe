# 프로젝트 규칙 (Rules)

- **파일 변경 전 승인 필수**: 소스 코드나 마크다운 파일을 실제로 수정/생성하기 전에, 변경될 코드 내용(Diff 또는 전체 코드)을 사용자에게 먼저 보여주고 진행 여부를 명시적으로 확인받아야 합니다.
- **초보자 맞춤형 변경 보고서 필수**: 파일 변경 승인을 요청할 때, 초보도 쉽게 이해할 수 있도록 친절한 보고서 형태로 작성해야 합니다. 보고서에는 반드시 다음 사항이 포함되어야 합니다:
  1. **수정하는 이유 (Why)**: 왜 이 변경이 필요한지, 기존 코드의 문제점이나 구현 목적을 설명합니다.
  2. **어떻게 변화하는지 (How & What)**: 변경 전과 비교하여 실제 기능이나 동작이 어떻게 바뀌는지 설명합니다.
- **대화보다 실행 우선 (코드 직접 기재 금지)**: 제안하거나 수정할 소스 코드는 대화방 본문에 직접 기재하지 않으며, 반드시 구현 계획서(implementation_plan.md) 내의 코드 블록으로만 작성해야 합니다. 대화방에서는 변경 맥락 보고서와 계획서 링크만 제공합니다.
- **계획서 제출 시 피드백 요청 필수**: 구현 계획서(implementation_plan.md)를 작성하거나 수정하여 승인을 요청할 때는, 관련 도구(write_to_file 등)의 ArtifactMetadata에서 RequestFeedback 옵션을 반드시 true로 설정하여 사용자 화면에 "Proceed" 승인 버튼이 나타나도록 강제해야 합니다.
- **계획서 내 설명용 주석 필수화**: 구현 계획서(implementation_plan.md)에 포함되는 코드 내용에는 각 라인/블록별로 왜 이 코드를 작성하는지 설명하는 상세한 주석을 달아야 합니다.
- **실제 반영 코드의 주석 청결화**: 실제 소스 코드 파일(Java 등)에 코드를 반영할 때는 계획서의 설명용 주석을 모두 제거하여 코드를 깔끔하게 유지합니다.
- **이모지 사용 절대 금지**: 모든 마크다운 문서, 구현 계획서, 소스 코드 주석 및 대화방 대답에서 이모지(Emoji)를 절대 사용하지 않습니다.
- **한국어 사용 절대 원칙**: 에이전트는 모든 대화방 대답 및 작성하는 마크다운 문서 내에서 무조건 한글(한국어)로만 작성해야 합니다.
- **개발 로그 작성 의무화**: 새로운 기능 추가, 규칙 수정, 아키텍처 개편 등 변경 작업이 완료되면 에이전트는 무조건 프로젝트 루트의 CHANGELOG.md 파일에 변경 이력을 날짜별로 누적하여 추가해야 합니다.
- **문서 조회 라우터 및 실수 사전 준수**: 에이전트는 불필요한 토큰 낭비를 방지하기 위해 파일 조회를 무작위로 수행하지 않으며, 반드시 [document_router.md](file:///Users/t2026-m0045/Downloads/cafe/docs/document_router.md)에 정의된 맵에 따라 지정된 문서만 읽어야 합니다. 또한 구현과 검토 시 [ai_common_mistakes.md](file:///Users/t2026-m0045/Downloads/cafe/docs/ai_common_mistakes.md)를 자가 점검표로 반드시 활용합니다.
