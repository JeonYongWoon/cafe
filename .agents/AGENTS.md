# 프로젝트 규칙 (Rules)

- **파일 변경 전 승인 필수**: 소스 코드나 마크다운 파일을 실제로 수정/생성하기 전에, 변경될 코드 내용(Diff 또는 전체 코드)을 사용자에게 먼저 보여주고 진행 여부를 명시적으로 확인받아야 합니다.
- **초보자 맞춤형 변경 보고서 필수**: 파일 변경 승인을 요청할 때, 초보자도 쉽게 이해할 수 있도록 친절한 보고서 형태로 작성해야 합니다. 보고서에는 반드시 다음 사항이 포함되어야 합니다:
  1. **수정하는 이유 (Why)**: 왜 이 변경이 필요한지, 기존 코드의 문제점이나 구현 목적을 설명합니다.
  2. **어떻게 변화하는지 (How & What)**: 변경 전과 비교하여 실제 기능이나 동작이 어떻게 바뀌는지 설명합니다.
- **대화보다 실행 우선 (코드 직접 기재 금지)**: 제안하거나 수정할 소스 코드는 대화방 본문에 직접 기재하지 않으며, 반드시 구현 계획서(`implementation_plan.md`) 내의 코드 블록으로만 작성해야 합니다. 대화방에서는 변경 맥락 보고서와 계획서 링크만 제공합니다.
- **계획서 제출 시 피드백 요청 필수**: 구현 계획서(`implementation_plan.md`)를 작성하거나 수정하여 승인을 요청할 때는, 관련 도구(`write_to_file` 등)의 `ArtifactMetadata`에서 `RequestFeedback` 옵션을 반드시 `true`로 설정하여 사용자 화면에 "Proceed" 승인 버튼이 나타나도록 강제해야 합니다.
- **계획서 내 설명용 주석 필수화**: 구현 계획서(`implementation_plan.md`)에 포함되는 코드 내용에는 각 라인/블록별로 왜 이 코드를 작성하는지 설명하는 상세한 주석을 달아야 합니다.
- **실제 반영 코드의 주석 청결화**: 실제 소스 코드 파일(Java 등)에 코드를 반영할 때는 계획서의 설명용 주석을 모두 제거하여 코드를 깔끔하게 유지합니다.
- **이모지 사용 절대 금지**: 모든 마크다운 문서, 구현 계획서, 소스 코드 주석 및 대화방 대답에서 이모지(Emoji)를 절대 사용하지 않습니다.
- **한국어 사용 절대 원칙**: 에이전트는 모든 대화방 대답 및 작성하는 마크다운 문서 내에서 무조건 한글(한국어)로만 작성해야 합니다.

# jeonscafe 프로젝트 공통 개발 컨벤션

## 1. 코드 컨벤션 (Code Convention)
* **Java 버전**: Java 17 표준 문법을 준수합니다.
* **들여쓰기(Indent)**: 4 Space 들여쓰기를 사용합니다. (Tab 대신 Space 권장)
* **Lombok 사용 가이드라인 (부작용 방지)**:
  롬복은 반복적인 상투적 코드를 줄여주어 편리하지만, 데이터가 유실되거나 서버가 다운되는 치명적인 부작용을 낳을 수 있어 사용 범위를 다음과 같이 제한합니다.
  
  1. **`@Setter` 사용 지양 (데이터 무단 수정 방지)**:
     * 클래스 레벨에 `@Setter`를 선언하면 프로젝트 전역에서 객체의 핵심 데이터를 아무런 제약 없이 마구 수정할 수 있게 되어, 데이터가 언제 왜 변했는지 파악하기 매우 어렵게 됩니다.
     * 따라서 데이터 변경은 단순 Setter가 아닌, 비즈니스 의미가 명확한 메서드(예: `chargePoint()`, `decreaseStock()`)를 직접 구현하여 사용합니다.
  
  2. **`@Data` 사용 금지 (JPA 양방향 순환 참조 방지)**:
     * `@Data`는 `@Getter`, `@Setter`, `@ToString` 등을 한 번에 생성해 주는 종합 선물 세트이지만, JPA 엔티티 객체에서 사용 시 큰 부작용이 있습니다.
     * 서로 참조하는 관계(예: 회원과 주문 목록)에서 `@Data`가 제공하는 `@ToString`이나 `@EqualsAndHashCode`를 자동 호출하면 **서로를 계속 호출하는 무한 루프에 빠져 서버가 멈추게(StackOverflowError) 됩니다.**
     * 엔티티 클래스에는 필요한 기능만 꺼내어 개별 애너테이션(`@Getter`, `@NoArgsConstructor` 등)으로 설정합니다.
  
  3. **`@AllArgsConstructor` 사용 제한 (매개변수 순서 오염 방지)**:
     * 모든 필드를 인자로 받는 생성자를 만들면, 클래스의 멤버 변수 선언 순서를 바꿨을 때 자바 컴파일 에러가 나지 않고 그대로 엉뚱한 필드에 데이터가 대입되어 대형 버그를 만듭니다.
     * 대신 꼭 필요한 필수 인자만 받는 생성자(`@RequiredArgsConstructor` 활용)를 쓰거나 `Builder 패턴`을 적극 활용합니다.

## 2. 네이밍 규칙 (Naming Convention)
* **패키지명**: 소문자 단수형 사용을 원칙으로 합니다. (예: `com.example.jeonscafe.order`)
* **클래스 및 인터페이스**: PascalCase를 사용합니다. (예: `OrderService`, `MenuRepository`)
* **메서드 및 변수**: camelCase를 사용합니다. (예: `chargePoint()`, `menuList`)
* **상수**: 영문 대문자와 Underbar(`_`)를 사용합니다. (예: `MAX_ORDER_LIMIT`)
* **DB 테이블 및 컬럼**: 
  * 테이블명은 snake_case 및 **복수형 명사**를 권장합니다. (예: `members`, `menus`, `orders`)
  * 특히 `order`는 SQL 예약어와 충돌하므로 `orders` 또는 `purchase`를 사용하여 충돌을 방지합니다.
  * 컬럼명은 snake_case 및 **단수형 명사**를 사용합니다. (예: `created_at`, `member_id`)

## 3. Git 및 커밋 컨벤션 (Git Convention)
* **브랜치 전략**: 
  * 기본 개발은 `main` 브랜치를 기준으로 하며, 기능 개발 시 `feature/{기능명}` 형식을 사용합니다. (예: `feature/charge-point`)
* **커밋 메시지 포맷**: `type: 내용` 형식을 따릅니다.
  * `feat:` 새로운 기능 추가
  * `fix:` 버그 수정
  * `docs:` 문서 수정 (README, CONVENTION 등)
  * `refactor:` 코드 리팩토링 (기능 변화 없음)
  * `chore:` 빌드 업무 수정, 패키지 매니저 설정 등 (코드 변화 없음)
  * *예시: `feat: 메뉴 목록 조회 API 구현`*

## 4. API 설계 규칙 (API Design Rules)
* **RESTful URI 규칙**:
  * 복수형 명사를 사용하며, 행위(동사)는 HTTP Method로 표현합니다.
  * 조회: `GET /menus`
  * 등록/충전/주문: `POST /orders`, `POST /points/charge`
* **공통 응답 JSON 포맷**: 모든 API는 클라이언트에게 일정한 구조로 응답을 내려주어야 합니다.

### 성공 응답 예시 (HTTP Status: 200 OK)
```json
{
  "success": true,
  "data": {
    "menuId": 1,
    "name": "아메리카노",
    "price": 4500
  },
  "error": null
}
```

### 실패 응답 예시 (HTTP Status: 400 Bad Request, 500 Internal Server Error 등)
```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "INSUFFICIENT_POINT",
    "message": "포인트 잔액이 부족합니다."
  }
}
```

## 5. 코드 리뷰 컨벤션 (Code Review Convention)

### 5.1. 요청자(Author) 가이드라인
* **자가 검토(Self-Review)**: PR(또는 변경 사항)을 제시하기 전에 변경된 코드 라인을 스스로 훑어보며 미처 정리하지 못한 디버깅용 코드나 임시 주석을 정리합니다.
* **변경 맥락 기술**: 변경 사유(Why)와 변경 사항(What/How)을 간결하게 작성하여 상대방(또는 에이전트)이 빠른 맥락 파악을 할 수 있도록 돕습니다.

### 5.2. 피드백 강도 표기법 (Pn Rule)
리뷰 의견의 중요도와 강도를 표시하기 위해 접두사에 **P1 ~ P5** 태그를 사용합니다. 에이전트가 코드 리뷰를 수행할 때도 이 기준에 맞춰 의견을 작성합니다.

* **P1: 필수 반영 (Must)**
  * 장애 유발 코드, 컴파일 에러, 기능 미작동 등 배포 전에 반드시 수정해야 하는 치명적인 사항입니다.
  * 예) `P1: 재고 차감 시 동시성 제어가 되지 않아 마이너스 재고가 발생할 수 있습니다.`
* **P2: 가급적 반영 (Should)**
  * 프로젝트의 핵심 컨벤션 위반(예: @Setter 사용 등)이나 심각한 비효율성 등 가급적 반영을 권장하는 사항입니다.
  * 예) `P2: 엔티티 클래스에는 롬복 사용 가이드라인에 따라 @Data 대신 @Getter와 @NoArgsConstructor를 사용해야 합니다.`
* **P3: 의견 제시 및 토론 (Ask/Discuss)**
  * 대안 설계나 기술적인 궁금증을 제시하고 함께 논의하고자 하는 사항입니다.
  * 예) `P3: 해당 예외 처리를 글로벌 핸들러로 위임하여 중복을 줄이는 것은 어떨까요?`
* **P4: 사소한 개선 및 제안 (Could)**
  * 오타 수정, 사소한 변수명 추천 등 반영 여부를 본인이 편하게 선택할 수 있는 가벼운 제안입니다.
  * 예) `P4: 변수명 menuList를 menus로 축약해도 의미 전달에 무리가 없을 것 같습니다.`
* **P5: 칭찬과 응원 (Praise)**
  * 깔끔한 코드 구성이나 좋은 설계 방향에 대한 긍정적인 피드백입니다.
  * 예) `P5: 예외 처리와 유효성 검증 로직이 매우 깔끔하게 잘 분리되었습니다.`
